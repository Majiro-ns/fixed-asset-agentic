## Agentic設計思想: 止まる → 聞く → 変わる

本システムの核心は「**Stop-first**」。AIが自律的に判断を制御する3段階の設計思想です。

```
  ┌──────────┐      ┌──────────┐      ┌──────────┐
  │  止まる   │ ──→ │   聞く   │ ──→ │  変わる   │
  │  (Stop)  │      │  (Ask)   │      │ (Change) │
  └──────────┘      └──────────┘      └──────────┘
   確信度が低い       不足情報を        フィードバックで
   → 自動判定を       具体的に提示      判定を修正
     一時停止         → Agentic Loop    → DIFF で証跡保存
```

| ステップ | 動作 | 技術実装 |
|---------|------|---------|
| **止まる** | 確信度 < 0.7 でパイプラインを一時停止 | `GUIDANCE` 判定 + `flags` に停止理由を記録 |
| **聞く** | 何が不足しているか、なぜ必要かを構造化して提示 | `missing_fields` + `why_missing_matters` |
| **変わる** | ユーザーの回答を受けて再判定。判定変更の全過程を記録 | `answers` → 再分類 → Before/After DIFF |

> **設計の本質**: 「すべてを自動化する」のではなく「判断を行う/止めるを自律的に選択する」。
> これが本システムの定義する Agentic AI です。

## なぜこのアプローチが優れているか

### 従来型ルールベース vs 本システム

| 比較軸 | ルールベース | 本システム（Stop-first AI） |
|--------|-------------|---------------------------|
| 判断方式 | 固定ルールで全件自動判定 | AIが文脈を読み、迷ったら停止 |
| 曖昧なケース | ルール外は誤判定 or エラー | GUIDANCE として人間に委任 |
| 根拠の提示 | ルールIDのみ | 判定理由 + 不足情報 + 法令エビデンス |
| ルール追加 | 開発者がコード修正 | ポリシーファイルで経理担当者が調整可能 |
| 監査対応 | ログ確認が困難 | Before/After DIFF で完全な証跡 |

### 手動判定 vs 本システム

| 比較軸 | 手動判定 | 本システム（Stop-first AI） |
|--------|---------|---------------------------|
| 処理時間 | 15分/件 | 5分/件（**67%削減**） |
| 判断のばらつき | 担当者依存 | 統一ルール + AI推論 |
| 月末・決算期 | 確認を省略しがち | GUIDANCE行のみ優先確認で効率化 |
| 引き継ぎ | 属人的な暗黙知 | エビデンス付きで判断根拠が残る |
| 誤判定リスク | 疲労・時間圧力で増大 | Stop-first で高リスク案件を自動検知 |

### Agentic AI の優位性（3つの柱）

1. **メタ認知**: AIが「自分の判断の不確実性」を認識し、停止判断を下せる
2. **対話的解決**: 単なるエラー返却ではなく、何が不足し、なぜ必要かを構造化して質問する
3. **完全な追跡性**: 判定の変更過程（Before/After DIFF）が監査証跡として保存される

## 競合との差別化

### 他のハッカソン作品との差別化要素

| 差別化要素 | 本システムの特徴 |
|-----------|----------------|
| **設計思想** | 「AIが止まる」ことを価値にする逆転の発想。全自動ではなく判断制御 |
| **実務密着** | 経理現場の「月末に検証する余裕がない」という現実課題から設計 |
| **監査対応** | DIFF表示による判定変更の完全なトレーサビリティ |
| **汎用性** | Stop-first パターンは医療診断・法務レビュー等、他ドメインにも適用可能 |

### Google Cloud サービスの活用

| サービス | 活用方法 | 効果 |
|---------|---------|------|
| **Gemini 3 Pro Preview** | `thinking_level=HIGH` で税務知識を注入した深い推論 | 複雑な判断の精度向上 |
| **Cloud Run** | API/UI のサーバーレスデプロイ | スケーラブルかつ低運用コスト |
| **Document AI** | PDF見積書からの高精度テキスト抽出 | 日本語帳票への対応力 |
| **Vertex AI Search** | 法令・通達の根拠を自動引用 | GUIDANCE の根拠を法令で補強 |

## セクション3: GUIDANCE -- 止まる→聞く→変わる→DIFF（0:40-2:10）

デモの核心。Agentic 5-Stepプロセスの全容を90秒で見せる。

### 画面操作
1. ドロップダウンから `demo04_guidance_ambiguous.json` を選択
2. 「Classify」をクリック
3. GUIDANCE結果を確認（黄色バッジ）
4. 「Agent Needs Information」パネルの内容を確認
5. クイックピックまたはフォームフィールドで回答を入力
6. 「Re-run Classification with Answers」をクリック
7. DIFFカードを表示・確認

### ナレーション

**【止まる】**

> **[0:40-0:55]**
> 次が核心です。今度は曖昧な見積書を入れます。古い設備の撤去と新しい設備の移設が混在した工事見積書。資産か経費か、人でも迷うケースです。
>
> GUIDANCE。AIが自ら判断を停止しました。「判断が割れるから、自分では決めない」という選択。これがStop-first設計の核心です。

**【聞く】**

> **[0:55-1:15]**
> 注目してください。AIは単に「わかりません」と言っているのではありません。何の情報が不足しているか、なぜその情報が判定に必要なのかを具体的に提示しています。
>
> 「工事の目的は原状回復か、それとも機能向上を伴うものか」。資本的支出か修繕費かの分かれ目を、AI自身が特定しています。

**【変わる】**

> **[1:15-1:40]**
> では回答を入力します。クイックピックで「機能向上を伴う工事」を選択します。これは現場で仕入先に確認する、あるいは契約書を見るという行為に相当します。再判定を実行します。
>
> 判定が変わりました。GUIDANCEからCAPITAL_LIKEへ。確信度も0.7から0.85に上昇しています。追加の情報を得たことで、AIが自信を持って判断できるようになりました。

**【DIFF】**

> **[1:40-2:10]**
> このDIFFカードをご覧ください。変更前後の判定、確信度、判定理由の差分が全て記録されています。
>
> なぜ判定が変わったのか。その過程が完全にトレースできます。これは監査証跡そのものです。経理担当者も、監査法人も、判断の根拠を後から確認できる。止まって、聞いて、変わった。その全てが残ります。

### 画面上の注目ポイント

**止まる・聞く**
- **判定バッジ**: `[GUIDANCE]`（黄色）
- **「Agent Needs Information」パネル**:
  - 不足フィールド一覧
  - 「Why This Matters」（重要性の説明）
  - 「What You Should Answer」

**変わる**
- クイックピックボタン: 「Repair/Maintenance」「Upgrade/New Asset」「Clear All Answers」
- フォームフィールド（ヘルプテキスト付き）

**DIFF**
- **DIFFカード**（再判定後に画面上部に表示）:
  - 判定: `GUIDANCE -> CAPITAL_LIKE`
  - 確信度: `0.70 -> 0.85`
  - トレース: `rerun_with_answers` ステップが追加

### 強調ポイント
- **最重要ビジュアル**: 黄色のGUIDANCEバッジが出る瞬間（間を取る）
- **Agenticモーメント**: AIが自律的に停止する瞬間
- **解決**: DIFFカードのBefore/After比較
- **監査**: 判定変更の完全な証跡
