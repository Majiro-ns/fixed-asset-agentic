## 1. 対象ユーザー

本システムは以下のユーザーを対象とする。

### 1.1 プライマリユーザー

| ユーザー区分 | 役割 | 利用頻度 | 主な利用シーン |
|-------------|------|----------|---------------|
| **中小企業 経理担当者** | 日常の経理処理、月次決算 | 日次〜週次 | 見積書の資産/費用判定 |
| **会計事務所スタッフ** | 顧問先の記帳代行、決算支援 | 日次 | 複数クライアントの一括処理 |
| **税理士** | 税務判断、決算申告 | 月次〜四半期 | 判断が割れる案件の確認 |

### 1.2 セカンダリユーザー

| ユーザー区分 | 役割 | 利用頻度 | 主な利用シーン |
|-------------|------|----------|---------------|
| **中小企業 経営者** | 経理業務の監督、最終承認 | 月次 | 大型設備投資の判断確認 |
| **CFO / 財務責任者** | 財務戦略、内部統制 | 四半期 | 監査対応、方針策定 |
| **内部監査担当者** | 処理の妥当性検証 | 年次 | 証跡レビュー |

## 2. 解決する課題

### 2.1 現状の課題（As-Is）

経理・税務の現場では、以下の構造的な課題が存在する。

#### 課題1: 判断の属人化
- **状況**: 固定資産/費用の判定は担当者の経験に依存
- **影響**: 担当者不在時に処理が滞る、引継ぎが困難
- **リスク**: 判断基準のばらつき、税務調査での説明困難

#### 課題2: 時間的プレッシャーによる見落とし
- **状況**: 月末・決算期は処理量が集中し、確認時間が不足
- **影響**: 判断が割れる項目（撤去・移設等）が他の行に埋もれる
- **リスク**: 誤処理、後からの訂正コスト増大

#### 課題3: 判断根拠の追跡困難
- **状況**: 「なぜこの処理になったか」の記録が残らない
- **影響**: 税務調査時に説明できない、過去の判断を参照できない
- **リスク**: 否認リスク、同じ論点での再検討コスト

#### 課題4: AI自動化の両刃の剣
- **状況**: 既存のAI自動化ツールは「すべて自動処理」が前提
- **影響**: 誤った判断が高速に通過してしまう
- **リスク**: 「AIが出したから正しい」という思考停止

### 2.2 解決後の姿（To-Be）

本システム導入により、以下の状態を実現する。

| 課題 | 解決後の姿 |
|-----|-----------|
| 判断の属人化 | 判断基準がシステムに組み込まれ、誰でも一定品質の処理が可能 |
| 時間的プレッシャー | 要確認項目（GUIDANCE）のみに集中でき、確認効率が向上 |
| 判断根拠の追跡困難 | 全ての判定に証跡（Evidence）が残り、後から追跡可能 |
| AI自動化の両刃の剣 | Stop-first設計により、AIが迷う場面では自動化が止まる |

## 3. 具体的な利用シナリオ

### シナリオ1: 月次決算での見積書処理

**ユーザー**: 中小企業 経理担当者（田中さん）

**背景**:
- 従業員50名規模の製造業
- 毎月末に10〜20件の見積書を処理
- 月末は残業が常態化

**従来のフロー**:
1. 見積書PDFを1件ずつ開く
2. 各明細行を目視で確認
3. 経験に基づき固定資産/費用を判定
4. 判断に迷ったら上司や税理士に相談
5. 仕訳を起票

**本システム導入後のフロー**:
1. 見積書PDFをアップロード
2. システムが自動で明細を抽出・判定
3. **CAPITAL_LIKE/EXPENSE_LIKE**: そのまま採用可能
4. **GUIDANCE**: 要確認項目として明示される
5. GUIDANCE項目のみ重点確認、必要に応じて追加情報を入力
6. 再判定結果を確認し、仕訳を起票

**効果**:
- 確認対象を要確認項目に絞り込み → 処理時間 60%削減
- 判断根拠が証跡として残る → 監査対応の効率化
- 「見落とし」リスクの低減 → 月末のストレス軽減

---

### シナリオ2: 大型設備投資の判定

**ユーザー**: CFO（山田さん）

**背景**:
- 年商20億円の建設会社
- 新規重機の導入を検討中（見積金額3,000万円）
- 付帯工事（据付、撤去、移設）の取扱いを確認したい

**利用フロー**:
1. 重機メーカーからの見積書PDFをアップロード
2. システムが明細を抽出
   - 本体: 2,500万円 → **CAPITAL_LIKE**
   - 据付工事: 300万円 → **GUIDANCE**（flags: 付帯工事）
   - 既設撤去: 150万円 → **GUIDANCE**（flags: 撤去）
   - 運搬費: 50万円 → **EXPENSE_LIKE**
3. GUIDANCE項目について、不足情報（Why Missing Matters）を確認
   - 「据付工事は取得価額に含めるか、事業供用後の資本的支出か？」
   - 「撤去は新規資産の取得に必要な除去費用か？」
4. 追加情報（契約書、工事概要）を入力して再判定
5. 最終判定と証跡をエクスポートし、税理士と共有

**効果**:
- 判断が割れる項目を事前に特定 → 税理士との相談が効率化
- 証跡付きで意思決定 → 税務調査への備え
- 「なぜこの処理にしたか」を後から説明可能

---

### シナリオ3: 会計事務所での複数クライアント処理

**ユーザー**: 会計事務所スタッフ（佐藤さん）

**背景**:
- 顧問先30社を担当
- 毎月合計100件以上の見積書を処理
- クライアントごとに判断基準が異なる（業種、規模、方針）

**利用フロー**:
1. クライアント別のポリシーファイル（policies/xxx.json）を選択
2. 見積書PDFをバッチアップロード
3. システムが一括で抽出・判定
4. ダッシュボードでGUIDANCE項目を一覧表示
5. 優先度の高いもの（金額大、判断が複雑）から順に確認
6. 処理結果をクライアント別にエクスポート

**効果**:
- クライアント固有の判断基準をポリシーとして管理
- バッチ処理により大量案件を効率的に処理
- GUIDANCE項目の優先度付けで重要案件に集中

### シナリオ4: 税務調査対応

**ユーザー**: 経理部長（鈴木さん）

**背景**:
- 税務調査で3年前の設備投資の処理根拠を問われた
- 当時の担当者は退職済み
- 「なぜ費用処理にしたのか」の説明が必要

**利用フロー**:
1. 当時の処理結果（証跡）をシステムから検索
2. 判定結果と根拠（Evidence）を確認
   - 判定: EXPENSE_LIKE（Confidence: 0.85）
   - 根拠: 「修繕費に該当（flags: repair_like, under_threshold）」
   - 金額: 18万円（少額資産基準未満）
3. 証跡レポートをエクスポート
4. 税務調査官に提示

**効果**:
- 過去の判断根拠を即座に確認可能
- 担当者が変わっても説明責任を果たせる
- 「AI任せ」ではなく、人間が確認した証跡として提示

### シナリオ5: 新人経理担当者の教育

**ユーザー**: 新人経理担当者（高橋さん）と指導担当（上司）

**背景**:
- 入社2ヶ月目の新人
- 固定資産と費用の区分を学習中
- 実務で判断を身につけたい

**利用フロー**:
1. 過去の見積書サンプルをシステムに入力
2. システムの判定結果と根拠を確認
3. 上司と一緒に「なぜこの判定になったか」を議論
   - 「撤去がGUIDANCEになった理由は？」
   - 「このflagsはどういう意味？」
4. 追加情報による再判定の変化を観察
5. 判断基準を実践的に学習

**効果**:
- 実例ベースで判断基準を学習
- 「なぜ」の説明がシステムに組み込まれている
- 上司の暗黙知をシステムが可視化

## 4. 導入効果

### 4.1 定量効果

| 効果項目 | 導入前 | 導入後 | 改善率 |
|---------|-------|-------|--------|
| 見積書処理時間（1件あたり） | 15分 | 5分 | **67%削減** |
| 月末残業時間（経理担当者） | 20時間 | 8時間 | **60%削減** |
| 判断ミス発生率 | 5% | 1%未満 | **80%以上削減** |
| 税務調査対応時間（1件あたり） | 2時間 | 30分 | **75%削減** |

※効果は想定値であり、実際の効果は導入環境により異なる

### 4.2 定性効果

| 効果項目 | 詳細 |
|---------|------|
| **属人化の解消** | 判断基準がシステムに組み込まれ、誰でも一定品質の処理が可能 |
| **心理的負担の軽減** | 「見落としたらどうしよう」という不安から解放 |
| **説明責任の強化** | 全ての判断に証跡が残り、後から説明可能 |
| **教育コストの削減** | 新人が実例ベースで判断基準を学習可能 |
| **監査対応の効率化** | 過去の判断根拠を即座に検索・提示可能 |

### 4.3 Stop-first設計による独自価値

従来のAI自動化ツールとの違い:

| 項目 | 従来のAI自動化 | 本システム（Stop-first） |
|------|---------------|------------------------|
| 設計思想 | 全てを自動化 | 判断を止める/行うを自律選択 |
| 不確実な判断 | 無理に断定 | GUIDANCE（要確認）を返す |
| ユーザーの役割 | 結果を受け入れる | 要確認項目を判断する |
| リスク | 誤判断が高速に通過 | 誤判断前に停止 |
| 証跡 | なし or 限定的 | 全判定に根拠を記録 |

|------|------|

2. **パイロットフェーズ**（1ヶ月）
   - 限定的な実データでの運用

3. **本番運用フェーズ**
   - 全見積書への適用
   - 継続的なポリシー改善

## 6. 想定される懸念と対応

### Q1: AIの判断を信用して大丈夫か？

**A**: 本システムはStop-first設計により、判断が割れる場面では自動化を停止し、人間の判断を促します。AIが断定する場面を意図的に制限しているため、「AI任せ」にはなりません。

### Q2: 既存の業務フローを大きく変える必要があるか？

**A**: 既存フローの「判定」部分を支援するツールであり、前後の業務（見積書受領、仕訳起票等）は変更不要です。

### Q3: 自社固有の判断基準に対応できるか？

**A**: ポリシーファイル（JSON）により、会社固有のキーワード、閾値、判断基準をカスタマイズ可能です。

### Q4: 過去のデータとの整合性は？

**A**: 本システムは「判定支援」ツールであり、最終判断は人間が行います。過去の判断基準と異なる提案がされた場合も、人間が最終決定します。

## 付録: 用語集

| 用語 | 説明 |
|------|------|
| **CAPITAL_LIKE** | 固定資産として計上すべき可能性が高い項目 |
| **EXPENSE_LIKE** | 費用として処理すべき可能性が高い項目 |
| **GUIDANCE** | 判断が割れるため、人間の確認が必要な項目 |
| **Evidence** | 判定の根拠となる情報（キーワード、金額、flags等） |
| **flags** | 判定に影響を与えた特徴（撤去、移設、修繕等） |
| **Stop-first** | 不確実な場面では自動化を止める設計思想 |
| **Policy** | 会社固有の判断基準を定義したJSON設定ファイル |
